Testing
Our test plan is based on the following five principles:
  1. Independence: Developers should write tests for their own code, because nobody understands a piece of code better than the person who wrote it. This will ensure that test methods properly test and suit the functional code it is meant to be testing.
  2. Incrementation: New functional code should be thoroughly tested at the same time it is being implemented and committed to the game repository. This eradicates the possibility of bugs in the project remaining undetected for a long period of time, when they may become harder to fix.
  3. Regression: All game code must be retested before new code can be committed to the game repository. This will ensure that adding new functionality isnâ€™t causing issues with our existing code, and ensure that the latest version of our game remains stable at all times.
  4. Completeness: Developers should comprehensively test new code under a wide range of scenarios and inputs using exhaustive test methods. This will ensure that code is of a high quality and will reduce the likelihood of code needing to be changed in the long run.
  5. Traceability: Developers should fill out our online test log for every commit made to the game repository. This will provide useful information on the effectiveness and completeness of test solution, as well as how well as a group we are doing in detecting and eradicating bugs in our work.
